# Phase01 주차별 핵심 발표 자료 디자인

## 목적

컴퓨터공학 4학년 수준이지만 MySQL 내부 동작과 인덱스를 처음 학습하는 청중을 대상으로, `phase01`의 1~5주차 학습이 하나의 성능 최적화 흐름으로 이어진다는 점을 설명한다.

발표가 끝났을 때 청중은 MySQL 성능 개선이 단순히 인덱스를 추가하는 일이 아니라, 실행 계획을 확인하고 읽기·정렬·그룹핑·테이블 접근 비용을 줄이는 과정임을 이해해야 한다.

## 산출물

- 형식: PowerPoint `.pptx`
- 분량: 14장
- 예상 발표 시간: 15~20분
- 화면 비율: 원본 Simple Light Mode 템플릿 유지
- 시각 기준: `artifact-template-simple-light-mode`의 retained reference를 복제·활용하며, 원본의 타이포그래피, 여백, 흑백 중심 팔레트와 반복 요소를 보존한다.
- 소스: `phase01`의 Markdown 7개와 포함된 실습 결과만 사용한다. 외부 수치나 검증되지 않은 성능 배수는 추가하지 않는다.

## 내러티브

발표의 중심 질문은 “MySQL은 쿼리를 처리할 때 어떻게 덜 읽고, 덜 정렬하고, 덜 접근하는가?”이다.

각 주차는 다음 주차의 필요성을 만든다.

1. 실행 계획을 봐야 MySQL의 선택을 알 수 있다.
2. B-Tree 인덱스로 읽는 범위를 줄일 수 있다.
3. 인덱스는 검색뿐 아니라 정렬과 조인 비용도 줄인다.
4. 인덱스 순서는 그룹핑과 중복 제거 방식에도 영향을 준다.
5. 필요한 컬럼을 인덱스가 모두 가지면 실제 테이블 접근까지 생략할 수 있다.

마지막에는 하나의 조회 쿼리에 다섯 주차의 개념을 적용해 전체 흐름을 연결한다.

## 슬라이드 구성

### 1. 표지

- 제목: `MySQL은 어떻게 덜 읽는가?`
- 부제: `Phase01 · 실행 계획부터 커버링 인덱스까지`
- 역할: 발표의 중심 질문 제시

### 2. 다섯 주차는 하나의 최적화 흐름이다

- 1주차부터 5주차까지의 누적 학습 흐름을 짧은 문장으로 제시한다.
- `실행 이해 → 검색 범위 축소 → 정렬 축소 → 집계 비용 축소 → 테이블 접근 생략`
- 역할: 주차별 내용이 분리된 주제가 아니라 연결된 과정임을 설명

### 3. 1주차 — SQL은 바로 실행되지 않는다

- 파서, 전처리기, 옵티마이저, 실행 엔진, InnoDB로 이어지는 쿼리 처리 흐름
- MySQL 엔진과 스토리지 엔진의 역할 구분
- 역할: 옵티마이저와 실행 계획을 학습해야 하는 이유 제시

### 4. 실행 계획은 옵티마이저의 판단 결과다

- `type`, `key`, `rows`, `Extra` 네 항목만 우선 확인
- 실습의 `ALL → ref`, `key: NULL → 인덱스명`, 예상 읽기 행 수 감소를 비교
- `EXPLAIN`은 추정, `EXPLAIN ANALYZE`는 실제 실행이라는 차이
- 역할: 이후 모든 주차에서 사용할 관찰 도구 확립

### 5. 2주차 — B-Tree는 필요한 범위만 찾는다

- 정렬된 인덱스의 루트·브랜치·리프 탐색
- 풀 테이블 스캔과 인덱스 레인지 스캔의 읽기 범위 차이
- 조회 성능과 쓰기·저장 공간 비용의 트레이드오프
- 역할: 인덱스가 빠른 이유를 자료구조 관점에서 설명

### 6. 복합 인덱스는 왼쪽부터 정렬된다

- `(region, age, joined_at)`의 사전식 정렬 구조
- `WHERE`의 작성 순서보다 인덱스 컬럼 순서가 중요함
- 효과적인 조건과 주의할 조건을 대표 예시만으로 비교
- 역할: 인덱스가 존재해도 쿼리와 맞지 않으면 범위를 줄이지 못한다는 점 설명

### 7. 3주차 — 옵티마이저는 가장 싼 계획을 고른다

- 비용 기반 최적화와 통계 기반 추정
- 인덱스가 있어도 결과가 많거나 테이블이 작으면 전체 스캔을 선택할 수 있음
- `ALL`과 `index`를 단순한 성능 등급이 아니라 접근 방식으로 설명
- 역할: “인덱스가 있으면 반드시 사용한다”는 오해 해소

### 8. 인덱스 순서가 맞으면 정렬을 생략할 수 있다

- 인덱스 정렬과 `Using filesort` 비교
- 조인에서 드라이빙 테이블 정렬과 조인 결과 전체 정렬의 차이
- 인덱스 정렬과 `LIMIT`이 결합되면 일찍 중단할 수 있음
- 역할: 인덱스가 검색 외 작업에도 영향을 준다는 점 확장

### 9. 4주차 — 먼저 줄이고 나중에 그룹핑한다

- `WHERE`는 그룹핑 전 행 필터링, `HAVING`은 그룹 결과 필터링
- 가능한 조건은 `WHERE`에서 먼저 줄이는 원칙
- 인덱스 순서 기반 그룹핑과 임시 테이블 기반 그룹핑 비교
- 역할: 집계 전에 데이터량을 줄이는 사고방식 설명

### 10. GROUP BY와 DISTINCT도 인덱스 순서를 이용한다

- 루스 인덱스 스캔의 그룹별 대표 값 탐색
- `Using index for group-by`, `Using temporary`의 의미
- `DISTINCT`는 SELECT 전체 컬럼 조합에 적용됨
- `Using index`는 GROUP BY 전용 메시지가 아니라 커버링 인덱스를 의미함
- 역할: 실행 계획 메시지를 정확히 구분

### 11. 5주차 — 세컨더리 인덱스는 PK를 저장한다

- InnoDB 클러스터링 인덱스 리프에는 실제 레코드가 저장됨
- 세컨더리 인덱스 리프에는 인덱스 키와 PK가 저장됨
- 일반 조회의 `세컨더리 인덱스 → PK → 실제 레코드` 흐름
- 역할: 커버링 인덱스가 제거하는 비용을 이해시키기 위한 구조 설명

### 12. 커버링 인덱스는 마지막 테이블 접근을 없앤다

- 쿼리에 필요한 컬럼이 모두 인덱스에 있을 때 실제 레코드 조회 생략
- `Using index`와 `Using index condition` 비교
- `SELECT *`가 커버링 인덱스에 불리한 이유
- 인덱스 크기와 쓰기 비용 때문에 필요한 쿼리에만 적용해야 함
- 역할: Phase01의 최종 최적화 단계 제시

### 13. 한 쿼리에 다섯 주차가 모두 들어 있다

- 대표 쿼리:

```sql
SELECT id, name
FROM member
WHERE region = 'SEOUL'
ORDER BY age, joined_at
LIMIT 10;
```

- 대표 인덱스: `(region, age, joined_at)`
- 실행 계획 확인, 범위 축소, 정렬 생략, LIMIT 조기 종료, 커버링 여부 판단을 순서대로 연결
- `name`이 인덱스에 없으므로 현재 구성은 커버링이 아님을 명시
- 역할: 주차별 개념을 하나의 판단 과정으로 통합

### 14. 좋은 인덱스는 실행 계획으로 검증한다

- 최종 체크리스트:
  1. 실제로 어떤 인덱스를 선택했는가?
  2. 읽는 행 수가 충분히 줄었는가?
  3. 별도 정렬이나 임시 테이블이 필요한가?
  4. 실제 테이블 접근을 줄일 수 있는가?
  5. 조회 이득이 쓰기·공간 비용보다 큰가?
- 역할: 발표의 중심 질문을 실무적인 확인 절차로 마무리

## 시각 디자인

- Simple Light Mode 원본을 시각적 단일 기준으로 사용한다.
- 흰 배경, 검정 중심 타이포그래피, 넓은 여백과 큰 제목을 유지한다.
- 슬라이드마다 하나의 주장만 전달하며, 본문은 최대 3~5개 핵심 항목으로 제한한다.
- 반복적인 카드형 UI는 사용하지 않는다.
- 구조 설명에 꼭 필요한 경우에만 단순 도형과 연결선을 사용한다.
- SQL과 `EXPLAIN` 값은 짧은 코드 블록 또는 큰 비교 문구로 표현한다.
- 이미지가 필요한 경우 원문에 포함된 실행 계획이나 구조 이미지만 사용하고, 가독성이 낮으면 핵심 정보를 텍스트로 재구성한다.
- 같은 레이아웃을 연속해서 반복하지 않고, 템플릿에 존재하는 레이아웃을 주차별 역할에 맞게 변주한다.

## 내용 정확성 원칙

- `type = ALL`, `Using filesort`, `Using temporary`를 무조건 나쁜 것으로 단정하지 않는다.
- `rows`는 InnoDB 통계에 기반한 추정값으로 설명한다.
- `Using filesort`가 반드시 디스크 정렬을 의미하지 않음을 명시한다.
- `Using index`와 `Using index condition`, `Using index for group-by`를 구분한다.
- 실습 환경에 따라 실행 계획이 달라질 수 있음을 발표 문맥에서 자연스럽게 반영한다.

## 검증 기준

- 최종 PPTX는 원본 템플릿을 변경하지 않고 별도 파일로 생성한다.
- 모든 슬라이드를 PNG로 렌더링해 한 장씩 전체 크기로 확인한다.
- 제목 줄바꿈, 본문 잘림, 요소 겹침과 슬라이드 밖 오버플로가 없어야 한다.
- 코드와 실행 계획 문구는 발표 화면에서 읽을 수 있는 크기를 유지한다.
- 14장이 1주차부터 5주차까지 논리적으로 누적되고, 마지막 예제와 체크리스트에서 하나로 연결되어야 한다.
- PowerPoint 구조 검사를 수행해 슬라이드 캔버스 밖 요소가 없는지 확인한다.
