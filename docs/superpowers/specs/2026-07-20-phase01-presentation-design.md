# Phase01 주차별 핵심 발표 자료 디자인

## 목적

컴퓨터공학 4학년 수준이지만 MySQL 내부 동작과 인덱스를 처음 학습하는 청중을 대상으로, `phase01`의 1~5주차 학습이 하나의 성능 최적화 흐름으로 이어진다는 점을 설명한다.

발표가 끝났을 때 청중은 MySQL 성능 개선이 단순히 인덱스를 추가하는 일이 아니라, 실행 계획을 확인하고 읽기·정렬·그룹핑·테이블 접근 비용을 줄이는 과정임을 이해해야 한다.

## 산출물

- 형식: PowerPoint `.pptx`
- 분량: 15장
- 예상 발표 시간: 15~20분
- 화면 비율: 원본 Simple Light Mode 템플릿 유지
- 시각 기준: `artifact-template-simple-light-mode`의 retained reference를 복제·활용하며, 원본의 타이포그래피, 여백, 흑백 중심 팔레트와 반복 요소를 보존한다.
- 소스: `phase01`의 Markdown 7개와 포함된 실습 결과만 사용한다. 외부 수치나 검증되지 않은 성능 배수는 추가하지 않는다.

## 내러티브

발표의 중심 질문은 “MySQL은 쿼리를 처리할 때 어떻게 덜 읽고, 덜 정렬하고, 덜 접근하는가?”이다.

각 주차는 다음 주차의 필요성을 만든다.

1. 실행 계획을 봐야 MySQL의 선택을 알 수 있다.
2. B-Tree 인덱스로 읽는 범위를 줄일 수 있다.
3. 인덱스는 검색뿐 아니라 정렬과 조인 비용도 줄인다.
4. 인덱스 순서는 그룹핑과 중복 제거 방식에도 영향을 준다.
5. 필요한 컬럼을 인덱스가 모두 가지면 실제 테이블 접근까지 생략할 수 있다.

마지막에는 하나의 조회 쿼리에 다섯 주차의 개념을 적용해 전체 흐름을 연결한다.

## 슬라이드 구성

### 1. 표지

- 제목: `MySQL은 어떻게 덜 읽는가?`
- 부제: `Phase01 · 실행 계획부터 커버링 인덱스까지`
- 역할: 발표의 중심 질문 제시

### 2. 다섯 주차는 하나의 최적화 흐름이다

- 1주차부터 5주차까지의 누적 학습 흐름을 짧은 문장으로 제시한다.
- `실행 이해 → 검색 범위 축소 → 정렬 축소 → 집계 비용 축소 → 테이블 접근 생략`
- 역할: 주차별 내용이 분리된 주제가 아니라 연결된 과정임을 설명

### 3. 1주차 — SQL은 선언이고, 실행 방법은 옵티마이저가 정한다

- SQL은 원하는 결과를 선언하며, 전체 스캔·인덱스 탐색·조인 순서 같은 물리적 실행 방법은 직접 지정하지 않는다는 개념
- 파서가 SQL을 토큰과 파스 트리로 바꾸고, 전처리기가 테이블·컬럼·권한을 검증하는 원리
- 옵티마이저가 통계와 예상 비용을 이용해 후보 실행 계획 중 하나를 선택하는 원리
- 실행 엔진은 선택된 계획을 해석하고, InnoDB는 실제 페이지와 레코드를 읽는 역할 구분
- 시각 흐름: `SQL → 파싱·검증 → 비용 비교 → 실행 계획 → InnoDB I/O`
- 역할: 같은 결과를 내는 SQL에도 여러 물리적 실행 방법이 존재하므로 실행 계획을 확인해야 한다는 필요성 제시

### 4. EXPLAIN은 네 번에 나눠 읽는다

- 1단계 `table`: 현재 행이 어느 테이블의 접근 계획인지 확인하며, 조인에서는 출력 행의 순서만으로 실제 실행 순서를 단정하지 않고 전체 계획과 함께 해석
- 2단계 `type`: `ALL`, `index`, `range`, `ref`, `eq_ref`, `const` 중 어떤 접근 방식을 선택했는지 확인
- 3단계 `possible_keys → key → key_len → ref`: 후보 인덱스, 실제 인덱스, 복합 인덱스에서 사용한 키 길이, 비교 대상을 순서대로 확인
- 4단계 `rows × filtered`와 `Extra`: 읽을 것으로 예상한 행 수와 다음 단계로 전달할 비율, 추가 필터·정렬·임시 테이블·커버링 여부 확인
- 실무 우선순위: `실제/예상 읽기량 → 접근 방식과 key → 추정 오차 → Extra의 추가 작업`
- `rows`는 반환 행 수가 아니라 접근 단계가 읽을 것으로 추정한 행 수이고, 조인에서는 반복 횟수까지 함께 봐야 함
- `ALL` 자체를 오류로 보지 않고 테이블 크기와 결과 비율을 함께 판단
- 시각 자료: `03_mysql_optimizer_query_and_order_01_full_table_scan_explain.png`를 기본 화면으로 사용하고 `type`, `key`, `rows`, `Extra`에 번호를 붙여 읽는 순서를 표시
- 역할: EXPLAIN의 많은 컬럼 중 무엇을 어떤 순서로 볼지 구체적인 판독 절차 제공

### 5. EXPLAIN ANALYZE는 추정과 실제의 차이를 보여준다

- `EXPLAIN`은 통계 기반 예상 계획이고 `EXPLAIN ANALYZE`는 쿼리를 실제 실행해 각 연산의 실측값을 기록한다는 차이
- 트리 출력은 가장 안쪽의 들여쓰기 깊은 연산부터 바깥쪽 부모 연산으로 읽음
- `cost=... rows=...`는 옵티마이저의 예상, `actual time=a..b rows=n loops=m`은 실제 첫 행·전체 행 처리 시간, 루프당 행 수와 반복 횟수를 나타냄
- `rows`와 `loops`를 함께 봐야 Nested Loop Join처럼 같은 연산이 여러 번 반복되는 비용을 놓치지 않음
- 예상 1건과 실제 수만 건처럼 추정 오차가 크면 통계, 데이터 분포, 조건 상관관계 또는 인덱스 설계를 점검
- 대표 비교:

```text
Table scan: actual rows=100000 → Filter: actual rows=1
Index lookup: actual rows=1
```

- `EXPLAIN ANALYZE`는 실제로 실행되므로 무거운 쿼리와 데이터 변경문에는 주의
- 시각 자료: 1주차 실습의 Table scan과 Index lookup 텍스트를 템플릿 타이포그래피로 재구성하고, 안쪽 연산에서 바깥쪽으로 읽는 방향을 표시
- 역할: 옵티마이저가 예상한 계획이 실제 데이터에서도 맞았는지를 검증하는 방법 설명

### 6. 2주차 — B+Tree는 탐색한 뒤 필요한 구간만 이어서 읽는다

- 문서에서는 B-Tree라고 부르지만, InnoDB 인덱스는 리프 노드가 연결된 B+Tree 계열이라는 정확한 개념
- 페이지 단위의 루트·브랜치·리프 구조와 높은 분기 계수 덕분에 적은 단계로 시작 위치를 찾는 원리
- 인덱스 레인지 스캔의 `seek → scan → row lookup` 세 단계
- 풀 테이블 스캔은 전체 페이지를 읽고, 레인지 스캔은 정렬된 리프에서 필요한 구간만 읽는 차이
- 인덱스 키가 커지면 페이지당 키 수가 줄고 인덱스 크기와 I/O가 증가할 수 있다는 원리
- `INSERT`, `DELETE`, 인덱스 컬럼 `UPDATE` 시 정렬 구조를 함께 유지해야 하는 조회·쓰기 트레이드오프
- 역할: 인덱스가 빠른 이유를 “정렬되어 있다”에서 끝내지 않고 페이지 탐색과 읽기량 관점으로 설명

### 7. 복합 인덱스는 왼쪽 컬럼부터 검색 범위를 만든다

- `(region, age, joined_at)`은 `region`으로 먼저 정렬되고, 같은 지역 안에서 `age`, 같은 나이 안에서 `joined_at`으로 정렬되는 사전식 정렬 원리
- 왼쪽 접두사 원칙: `region` 또는 `region, age`는 시작 위치를 잡을 수 있지만 `age`만으로는 전체 인덱스의 연속 범위를 만들기 어려움
- 동등 조건 뒤에 범위 또는 정렬 컬럼을 두는 실무적 출발점: `region = ?` 다음 `ORDER BY age, joined_at`
- 첫 범위 조건 뒤의 컬럼은 탐색 범위를 추가로 줄이는 데 제한이 있지만 필터링이나 커버링에는 도움을 줄 수 있음
- `LIKE 'abc%'`는 시작·끝 범위를 만들 수 있지만 `LIKE '%abc'`는 시작점을 알 수 없다는 정렬 기반 원리
- 인덱스 컬럼을 함수로 감싸거나 타입을 다르게 비교하면 기존 정렬을 그대로 사용하기 어려운 SARGability 개념
- 역할: 인덱스가 존재하는 것보다 쿼리 조건과 인덱스 정렬 순서가 맞는 것이 중요하다는 점 설명

### 8. 3주차 — 옵티마이저는 인덱스 유무가 아니라 총비용을 비교한다

- 비용 기반 옵티마이저가 테이블 크기, 값 분포, 예상 행 수, 랜덤 I/O와 정렬 비용을 이용해 후보 계획을 비교하는 원리
- 카디널리티는 서로 다른 값의 수, 조건의 선택성은 전체 데이터에서 결과를 얼마나 줄이는지를 나타낸다는 개념
- 인덱스 조회는 적은 행을 찾을 때 유리하지만 결과가 많으면 레코드마다 발생하는 테이블 접근 때문에 전체 스캔보다 비쌀 수 있음
- `type = ALL`은 테이블 전체, `type = index`는 더 작은 인덱스 전체를 읽는 접근 방식이라는 차이
- 작은 테이블이나 대부분의 행을 반환하는 쿼리에서는 전체 스캔이 합리적일 수 있음
- 역할: “인덱스가 있는데 왜 사용하지 않았는가?”를 비용과 선택성으로 설명

### 9. 인덱스 순서가 맞으면 정렬 결과를 바로 흘려보낼 수 있다

- `ORDER BY` 순서가 인덱스와 맞으면 이미 정렬된 리프 노드를 순서대로 읽기만 하는 원리
- 맞지 않으면 MySQL이 정렬 키와 결과 데이터를 소트 버퍼에 모아 Filesort를 수행하며, 버퍼를 넘으면 디스크 병합 비용이 생길 수 있음
- `Using filesort`는 디스크 사용 여부가 아니라 별도 정렬 알고리즘을 수행했다는 의미
- Nested Loop Join에서 먼저 읽는 드라이빙 테이블과 반복 조회되는 드리븐 테이블의 역할
- `드라이빙 테이블만 정렬 → 조인`은 `조인 결과 전체 → 임시 테이블 → 정렬`보다 중간 데이터가 적을 수 있음
- 인덱스 정렬은 스트리밍하면서 `LIMIT`을 채우면 중단할 수 있지만 Filesort는 결과를 내기 전 선행 작업이 필요하다는 차이
- 역할: 인덱스가 검색뿐 아니라 정렬 대상 수, 첫 응답 시간과 LIMIT 효과에도 영향을 준다는 점 설명

### 10. 4주차 — 그룹핑 전에 행 수를 먼저 줄인다

- 논리 처리 단계에서 `WHERE`는 개별 행을 먼저 제거하고, `GROUP BY`는 남은 행을 그룹으로 모으며, `HAVING`은 집계된 그룹을 필터링한다는 개념
- `COUNT(*) > 10`처럼 집계 결과에 대한 조건은 `HAVING`이 필요하지만, `region = 'SEOUL'`처럼 행 단계에서 판단 가능한 조건은 `WHERE`로 앞당길 수 있음
- 인덱스가 그룹 컬럼 순서와 맞으면 같은 값이 연속되어 있어 순서대로 읽으며 그룹 경계를 판단할 수 있음
- 인덱스 순서가 맞지 않으면 키별 집계 상태를 저장하기 위한 내부 임시 테이블이 필요할 수 있음
- MySQL 8.0의 `GROUP BY`는 결과 정렬을 보장하지 않으므로 정렬이 필요하면 `ORDER BY`를 명시해야 함
- 역할: 집계 쿼리도 “먼저 읽는 데이터량을 줄인다”는 동일한 최적화 원칙을 따른다는 점 설명

### 11. GROUP BY와 DISTINCT는 같은 값을 모으는 비용을 가진다

- 타이트 인덱스 스캔은 그룹에 속한 인덱스 항목을 연속해서 읽고, 루스 인덱스 스캔은 그룹별 첫 값이나 마지막 값만 읽고 다음 그룹으로 건너뛰는 차이
- `(member_id, reservation_time)`에서 회원별 `MIN(reservation_time)`은 각 그룹의 첫 리프 항목만 읽을 수 있다는 원리
- `Using index for group-by`는 루스 인덱스 스캔, `Using temporary`는 중간 그룹 상태 저장 가능성을 나타냄
- 단순 `DISTINCT`는 선택한 전체 컬럼 조합의 중복을 제거하며 GROUP BY와 비슷한 방식으로 처리될 수 있음
- `COUNT(DISTINCT column)`은 집계 내부에서 별도의 중복 제거 상태가 필요할 수 있음
- `Using index`는 GROUP BY 전용 메시지가 아니라 필요한 컬럼을 인덱스만으로 읽은 커버링 인덱스를 의미함
- 역할: 집계 방식과 실행 계획 메시지를 연결하고 세 가지 `Using index...` 표현의 차이를 정리

### 12. 5주차 — InnoDB의 실제 레코드는 PK 순서의 리프에 있다

- InnoDB 클러스터링 인덱스는 PK로 정렬되고, 리프 노드에 실제 레코드 전체를 저장한다는 원리
- 세컨더리 인덱스 리프에는 물리 주소 대신 `세컨더리 키 + PK`를 저장해 레코드 이동 시 모든 보조 인덱스 주소를 수정하지 않도록 함
- 일반적인 세컨더리 인덱스 조회는 `세컨더리 B+Tree 탐색 → PK 획득 → 클러스터링 B+Tree 재탐색 → 레코드`의 이중 탐색
- PK가 길면 모든 세컨더리 인덱스 리프도 커지므로 PK 크기가 전체 인덱스 비용에 영향을 줌
- 역할: 실제 레코드 접근이 단순한 한 번의 포인터 이동이 아니며, 커버링 인덱스가 어떤 비용을 제거하는지 설명

### 13. 커버링 인덱스는 두 번째 B+Tree 탐색을 생략한다

- 필터링·조인·정렬·반환에 필요한 컬럼이 모두 인덱스에 있으면 인덱스 리프만으로 결과를 만들 수 있다는 개념
- InnoDB 세컨더리 인덱스에는 PK가 자동 포함되므로 `INDEX(emp_name)`만 있어도 `emp_name, emp_no`를 반환하는 쿼리는 커버링될 수 있음
- `Using index`는 실제 레코드 조회를 생략한 커버링, `Using index condition`은 인덱스에서 조건을 먼저 검사하지만 통과한 행의 실제 레코드는 읽는 ICP라는 차이
- 시각 자료: `05_covering_index_not_applied.png`와 `05_covering_index_applied.png`에서 `Extra`를 같은 위치로 맞춰 `Using index condition`과 `Using index`를 직접 비교
- `SELECT *`는 필요한 컬럼 범위를 넓혀 커버링 가능성을 낮춤
- 조회 컬럼을 무조건 인덱스에 추가하면 인덱스 페이지가 커지고 쓰기와 캐시 비용이 증가하므로 반복 빈도와 읽기 절감 효과를 함께 판단해야 함
- 역할: Phase01의 최종 최적화인 “테이블 접근 자체를 제거한다”는 원리와 트레이드오프 제시

### 14. 한 쿼리에 다섯 주차가 모두 들어 있다

- 대표 쿼리:

```sql
SELECT id, name
FROM member
WHERE region = 'SEOUL'
ORDER BY age, joined_at
LIMIT 10;
```

- 대표 인덱스: `(region, age, joined_at)`
- 1주차: `EXPLAIN ANALYZE`로 실제 선택한 `type`, `key`, `rows`, 정렬 여부 확인
- 2주차: `region = 'SEOUL'`로 복합 인덱스의 연속 범위를 탐색
- 3주차: 같은 지역 안에서 `age, joined_at` 순서대로 읽어 Filesort를 피하고 `LIMIT 10`에서 중단
- 4주차: 이 쿼리에는 그룹핑이 없지만, 같은 왼쪽 접두사 원리가 `GROUP BY region, age`에도 적용됨을 연결
- 5주차: `name`이 인덱스에 없으므로 실제 레코드 조회가 필요하고 현재 구성은 커버링이 아님
- 인덱스에 `name`을 추가하면 커버링 가능성이 생기지만 인덱스 크기와 쓰기 비용이 늘어난다는 설계 판단까지 제시
- 역할: 주차별 개념을 하나의 판단 과정으로 통합

### 15. 좋은 인덱스는 실행 계획으로 검증한다

- 최종 체크리스트:
  1. 실제로 어떤 인덱스를 선택했는가?
  2. 읽는 행 수가 충분히 줄었는가?
  3. 별도 정렬이나 임시 테이블이 필요한가?
  4. 실제 테이블 접근을 줄일 수 있는가?
  5. 조회 이득이 쓰기·공간 비용보다 큰가?
- 역할: 발표의 중심 질문을 실무적인 확인 절차로 마무리

## 시각 디자인

- Simple Light Mode 원본을 시각적 단일 기준으로 사용한다.
- 흰 배경, 검정 중심 타이포그래피, 넓은 여백과 큰 제목을 유지한다.
- 슬라이드마다 하나의 주장만 전달하며, 본문은 최대 3~5개 핵심 항목으로 제한한다.
- 반복적인 카드형 UI는 사용하지 않는다.
- 구조 설명에 꼭 필요한 경우에만 단순 도형과 연결선을 사용한다.
- SQL과 `EXPLAIN` 값은 짧은 코드 블록 또는 큰 비교 문구로 표현한다.
- 이미지가 필요한 경우 원문에 포함된 실행 계획이나 구조 이미지만 사용하고, 가독성이 낮으면 핵심 정보를 텍스트로 재구성한다.
- 같은 레이아웃을 연속해서 반복하지 않고, 템플릿에 존재하는 레이아웃을 주차별 역할에 맞게 변주한다.
- 각 주차의 첫 슬라이드는 핵심 개념과 내부 원리를 설명하고, 두 번째 슬라이드는 SQL 또는 실행 계획 신호를 이용해 원리가 실제 쿼리에 어떻게 나타나는지 보여준다.
- 상세 설명은 작은 글씨로 나열하지 않고, 한 슬라이드당 `핵심 주장 1개 + 원리 2~3단계 + 대표 예시 1개`로 편집한다.
- EXPLAIN 이미지는 전체 IDE 화면을 그대로 축소하지 않고, 필요한 결과 그리드 영역만 사용한다. 컬럼명과 값이 발표 화면에서 읽히도록 자르고 확대하며, 원본 픽셀을 불필요하게 늘리지 않는다.
- EXPLAIN 판독 슬라이드에서는 원본 스크린샷 위에 최소한의 번호와 강조선만 추가하며, 원본에 없는 실행 결과를 이미지처럼 꾸며 만들지 않는다.

## 내용 정확성 원칙

- `type = ALL`, `Using filesort`, `Using temporary`를 무조건 나쁜 것으로 단정하지 않는다.
- `rows`는 InnoDB 통계에 기반한 추정값으로 설명한다.
- `Using filesort`가 반드시 디스크 정렬을 의미하지 않음을 명시한다.
- `Using index`와 `Using index condition`, `Using index for group-by`를 구분한다.
- 실습 환경에 따라 실행 계획이 달라질 수 있음을 발표 문맥에서 자연스럽게 반영한다.

## 검증 기준

- 최종 PPTX는 원본 템플릿을 변경하지 않고 별도 파일로 생성한다.
- 모든 슬라이드를 PNG로 렌더링해 한 장씩 전체 크기로 확인한다.
- 제목 줄바꿈, 본문 잘림, 요소 겹침과 슬라이드 밖 오버플로가 없어야 한다.
- 코드와 실행 계획 문구는 발표 화면에서 읽을 수 있는 크기를 유지한다.
- 15장이 1주차부터 5주차까지 논리적으로 누적되고, 마지막 예제와 체크리스트에서 하나로 연결되어야 한다.
- PowerPoint 구조 검사를 수행해 슬라이드 캔버스 밖 요소가 없는지 확인한다.
